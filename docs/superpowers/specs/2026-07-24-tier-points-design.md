# 등급제/포인트 시스템 (1단계: 포인트·레벨업·일일 사용량 제한) 설계

## 배경 및 목적

로그인(카카오)까지 만들어졌으니, 그 위에 등급제를 얹는다. 이번 스펙은 4단계 전체 로드맵 중 **1단계만** 다룬다:

1. **포인트/레벨업/일일 사용량 제한** ← 이번 스펙
2. 번호 저장 + 마이페이지 (월별/주별 조회)
3. 당첨 확인 → "뽑기의 신" 승급
4. 구독/결제 (마스터 등급 즉시 부여)

2~4번은 각각 별도 스펙으로 이후에 진행한다. "뽑기의 신" 등급은 이번 스펙에서 데이터상 자리(enum)만 마련해두고, 실제로 도달 가능하게 만드는 건 3단계 몫이다.

## 범위

**포함:**
- 타로/번호생성 사용, 출석, 연속 출석에 따른 포인트 적립
- 누적 포인트 기준 자동 승급 (초심자 → 견습생 → 고수 → 마스터)
- 등급별 일일 사용 횟수 제한 (타로/번호생성 각각 별도 카운트)
- 타로/번호생성은 로그인 필수로 전환 (통계/회차조회는 그대로 비로그인 자유 열람)
- 타로 뽑기도 서버에서 사용횟수를 강제할 수 있도록 서버 게이트 추가 (현재 타로는 순수 프론트 계산이라 서버를 전혀 안 거침)

**제외:**
- 번호 저장, 마이페이지, 당첨 확인, 구독/결제 (다음 스펙들)
- "뽑기의 신" 등급에 실제로 도달하는 로직

## 등급 & 일일 사용량

| 등급 | 타로/일 | 번호생성/일 | 누적 포인트 구간 |
|---|---|---|---|
| 초심자 (기본) | 1회 | 1회 | 0 ~ 49P |
| 견습생 | 2회 | 2회 | 50 ~ 149P |
| 고수 | 3회 | 3회 | 150 ~ 349P |
| 마스터 | 5회 | 5회 | 350P ~ |
| 뽑기의 신 | 무제한 | 무제한 | (이번 단계에서 도달 불가 — 3단계에서 정의) |

번호생성은 가중치 모드/랜덤 모드를 합쳐서 하나의 횟수로 카운트한다 (모드 상관없이 "번호생성 1회" = 1 사용). 한 번의 `/api/generate` 호출에서 세트를 여러 개(1~10) 요청해도 호출 자체를 1회 사용으로 카운트한다 (세트 개수만큼 차감하지 않음).

등급은 별도 컬럼으로 저장하지 않고 **누적 포인트로부터 항상 계산**한다 (등급이 포인트와 따로 놀 여지를 원천 차단).

## 포인트 적립 규칙

- 타로 1회 사용 시 +1P
- 번호생성 1회 사용 시 +1P
- 로그인 상태로 하루 중 처음 사이트 방문 시 "출석" +2P (페이지 종류 무관 — 통계/회차조회만 봐도 인정)
- 7일 연속 출석 달성 시 보너스 +10P. 하루라도 방문을 거르면 연속 카운트는 0으로 초기화된다.

## 아키텍처

### 백엔드 (새 패키지 `com.lottopredictor.backend.progress`)

- **`User` 엔티티 확장**: `totalPoints`(int, 기본 0), `currentStreak`(int, 기본 0), `lastActiveDate`(date, nullable) 컬럼 추가. 마이그레이션 `db/migrations/0004_add_progress_to_users.sql`.
- **`Feature`**: `TAROT`, `GENERATE` enum
- **`Tier`**: `BEGINNER`, `APPRENTICE`, `EXPERT`, `MASTER`, `LOTTO_GOD` enum + 한글 라벨
- **`TierPolicy`**: 순수 함수 모음 — 누적 포인트 → `Tier`, `Tier`+`Feature` → 일일 한도, 다음 등급까지 필요 포인트
- **`DailyUsage` 엔티티**: `(user_id, usage_date, feature)` 복합키 + `count`. 마이그레이션 `db/migrations/0005_create_daily_usage.sql`.
- **`UsageService`**: 핵심 로직
  - `recordVisit(User)`: `lastActiveDate`가 오늘이 아니면 출석 +2P, 연속출석일수 갱신(어제 방문이면 +1, 아니면 1로 리셋), 7일 도달 시 +10P 보너스, `lastActiveDate` = 오늘로 갱신
  - `consume(User, Feature)`: 오늘 해당 `Feature` 사용량이 현재 등급의 일일 한도 미만이면 사용량 +1, +1P, 등급은 자동으로 재계산됨(별도 갱신 로직 불필요) → 성공 반환. 한도 도달 시 실패(초과) 반환, 아무것도 변경하지 않음
- **`ProgressResponse`** DTO: `tier`(한글 라벨), `totalPoints`, `pointsToNextTier`(마스터면 null), `tarotUsage`{used, limit}, `generateUsage`{used, limit}
- **엔드포인트**:
  - `GET /api/progress/me` (`@AuthPrincipal` 필수): `recordVisit` 호출(출석 처리) 후 `ProgressResponse` 반환. 프론트가 앱 로드 시 1회 호출 — 출석 체크와 현재 상태 조회를 겸함
  - `POST /api/progress/tarot-usage` (`@AuthPrincipal` 필수): `consume(user, TAROT)` 시도 → 성공 시 갱신된 `ProgressResponse` 200, 실패(한도 초과) 시 429
- **`/api/generate` 수정**: `@AuthPrincipal` 필수로 전환 (기존엔 비로그인도 호출 가능했음 — 동작 변경). 번호 생성 전에 `consume(user, GENERATE)` 시도, 한도 초과면 429, 통과하면 기존대로 생성 후 결과 반환

### 프론트엔드

- **`app/contexts/ProgressContext.tsx`** (신규): `AuthContext`와 비슷한 패턴. 로그인 상태가 되면 `GET /api/progress/me`를 1회 호출해 등급/포인트/오늘 사용량을 상태로 들고 있음. `layout.tsx`에서 `AuthProvider` 안쪽에 중첩해서 사이트 전역에 마운트 (어느 페이지를 봐도 출석이 인정되도록)
- **`/tarot` 페이지**: 로그인 안 돼있으면 카드 뽑기 대신 "로그인이 필요해요" 안내 + 로그인 버튼 표시(페이지 자체는 그대로 보여줌). 로그인 상태면 카드를 뽑기 직전에 `POST /api/progress/tarot-usage` 호출 → 성공해야 기존 드래그/공개 애니메이션 진행, 429면 "오늘 사용 횟수를 다 쓰셨어요. 등급을 올리면 더 뽑을 수 있어요" 메시지 표시
- **`/generate` 페이지**: 로그인 안 돼있으면 생성 폼 대신 로그인 안내. 로그인 상태면 기존 `generateNumbers` 호출에 `Authorization` 헤더 추가(기존 `lib/api.ts`는 토큰을 안 보내고 있었음 — 수정 필요), 429 응답 시 동일한 한도 초과 메시지 표시
- 두 페이지 모두 `ProgressContext`의 오늘 사용량/한도를 이용해 "오늘 남은 횟수: n/m (OO 등급)" 정도의 작은 안내를 뽑기 버튼 근처에 표시

## 에러 처리

- 미로그인 상태로 타로/번호생성 시도 → 프론트에서 애초에 버튼 대신 로그인 안내를 보여주므로 백엔드까지 갈 일이 없음. 그래도 직접 API를 호출하는 경우 `@AuthPrincipal`이 401 처리
- 일일 한도 초과 → 429, 프론트는 위 메시지로 안내 (에러가 아니라 정상적인 상태이므로 콘솔 에러로 취급하지 않음)
- 동시 요청으로 한도를 넘겨 소진하는 경쟁 상태(같은 유저가 여러 탭에서 동시에 클릭)는 이번 스펙에서는 감수한다 — 하루 최대 몇 회 차이가 사용성에 큰 영향을 주지 않는 취미 프로젝트 규모라 낙관적 처리로 충분

## 테스트

- 백엔드: `TierPolicy`(포인트→등급 매핑, 등급별 한도, 다음 등급까지 포인트) 순수 함수 단위 테스트. `UsageService`(한도 내 소비 성공, 한도 도달 시 실패 및 상태 불변, 출석 최초 1회만 적립, 연속출석 갱신, 하루 건너뛰면 연속 리셋, 7일 보너스) 단위 테스트 — `UserRepository`/`DailyUsageRepository`는 Mockito로 목 처리
- 프론트: `ProgressContext`는 기존 `AuthContext`와 동일한 컨벤션으로 자동 테스트 없이 타입체크 + 브라우저 수동 확인 (로그인 후 상태 조회, 한도 도달 시 메시지, 미로그인 시 안내 문구)
