# 로또 번호 예측 웹앱 — 1단계 MVP 설계

## 배경 및 목적

동행복권 사이트에 공개된 로또 6/45 역대 당첨번호를 자동으로 수집(크롤링)해서 DB에 쌓고, 이 데이터를 바탕으로 번호를 생성해주는 웹앱을 만든다. 장기적으로는 연금복권 추가, 타로점 등 다른 콘텐츠 확장, 구독 또는 광고 기반 수익화까지 고려하고 있으나, 이번 1단계 설계는 **로또 6/45만을 대상으로 한 핵심 기능 MVP**로 범위를 한정한다. 회원가입, 결제, 광고는 1단계에 포함하지 않는다.

이 프로젝트는 기존 언리얼 엔진 슈터 포트폴리오 프로젝트(`SecondSingleProject`)와는 완전히 별개이며, `D:\LottoPredictor`에 새로운 저장소로 시작한다.

### "예측"에 대한 중요한 전제

로또는 회차마다 완전히 독립적인 추첨이라, 과거 당첨 이력이 미래 당첨 확률에 통계적으로 영향을 주지 않는다. 따라서 이 앱의 "번호 생성"은 실제 예측력이 있는 것이 아니라, **출현 빈도 등 통계를 가중치로 삼은 랜덤 생성**이다. 다른 로또 앱들도 대부분 이런 방식이며, 재미 요소로 통계를 보여주는 것이 목적이다. 사용자에게도 이 점을 오해시키지 않도록 UI 문구에 유의한다 (예: "AI가 예측한 번호"보다는 "통계 기반 추천 번호" 같은 표현).

## 아키텍처 개요

Next.js(TypeScript) 단일 프로젝트로 프론트엔드와 API 라우트를 함께 구성한다. 데이터베이스는 PostgreSQL(Supabase 사용, 무료 티어로 시작). 배포는 Vercel을 사용하고, Vercel Cron이 매주 토요일 밤(로또 추첨 이후) API 라우트를 호출해 최신 회차를 크롤링한다.

```
[Vercel Cron] --매주 토요일 밤--> [/api/cron/crawl] --fetch--> [동행복권 공개 데이터] --upsert--> [Supabase DB]
[사용자] --> [Next.js 페이지] --> [/api/generate, /api/stats, /api/draws] --> [Supabase DB]
```

이 구성을 선택한 이유:
- Next.js 하나로 프론트/백엔드를 같은 언어(TypeScript)로 처리해 초기 개발 속도가 빠르다.
- Vercel + Supabase 조합은 둘 다 무료로 시작 가능하고, 나중에 트래픽이 늘거나 회원/결제 기능을 붙일 때 자연스럽게 확장된다.
- Vercel Cron은 별도 서버 없이 정기 작업을 처리할 수 있어 인프라 관리 부담이 적다.

## 데이터베이스 스키마

`lotto_draws` 테이블 하나로 시작한다.

| 컬럼 | 타입 | 설명 |
|---|---|---|
| draw_no | integer, PK | 회차 번호 |
| draw_date | date | 추첨일 |
| num1 ~ num6 | smallint | 당첨번호 6개 |
| bonus_num | smallint | 보너스 번호 |
| created_at | timestamp | 저장 시각 (기본값: 현재 시각) |

`draw_no`가 기본키(유니크)이므로, 크롤러가 같은 회차를 다시 수집해도 upsert로 처리되어 중복 저장되지 않는다.

## 크롤러

동행복권이 공개 제공하는 회차별 조회 API(`getLottoNumber`)를 이용해 데이터를 가져온다. HTML을 파싱하는 스크래핑 방식보다 안정적이고, 요청 빈도도 회차당 1회로 매우 낮아 서버에 부담을 주지 않는다.

동작 순서:
1. DB에서 저장된 가장 최신 회차 번호를 확인한다.
2. 그다음 회차부터 최신 회차까지 순서대로 데이터를 요청한다. (최초 실행 시에는 1회차부터 전체를 백필한다.)
3. 특정 회차 요청이 실패하거나 응답 형식이 예상과 다르면, 해당 회차는 건너뛰고 로그만 남긴다. 크롤러 전체가 중단되지 않으며, 다음 실행 때 다시 시도된다.
4. `/api/cron/crawl` 라우트는 Vercel Cron 외의 경로로 호출되지 않도록 시크릿 헤더로 보호한다.

## 번호 생성 로직

`/api/generate` 엔드포인트에서 처리한다.

1. DB에 저장된 전체 회차 데이터에서 1~45번 각 번호의 출현 빈도를 계산한다.
2. 출현 빈도를 가중치로 사용해, 가중 랜덤 샘플링(비복원 추출)으로 서로 다른 6개 번호를 뽑는다.
3. 가중치 없는 "완전 랜덤" 모드도 옵션으로 제공해, 사용자가 두 방식을 비교해볼 수 있게 한다.
4. DB에 데이터가 없는 경우(최초 배포 직후 등)에는 자동으로 완전 랜덤 모드로 대체(fallback) 동작한다.

## 프론트엔드 페이지 구성

- **홈**: 번호 생성 버튼을 누르면 결과 세트가 표시된다. 여러 세트를 한 번에 생성할 수 있다.
- **통계**: 번호별 출현 빈도, 최근 오랫동안 나오지 않은 번호 등을 시각화해서 보여준다.
- **회차 조회**: 역대 당첨번호를 목록으로 보거나 회차/날짜로 검색할 수 있다.

## 에러 처리

- 크롤러 실패는 기존 서비스 동작에 영향을 주지 않는다 (독립된 API 라우트에서 처리되며, 실패해도 다음 주기에 재시도).
- 번호 생성 시 DB가 비어있으면 완전 랜덤으로 자동 폴백한다.
- 동행복권 응답 형식이 변경되는 경우를 대비해, 파싱 실패는 예외로 처리하고 로그를 남긴다.

## 테스트 범위

- 가중 랜덤 번호 생성 로직: 단위 테스트로 분포가 합리적인지 확인 (예: 특정 번호에 가중치를 높였을 때 실제로 더 자주 뽑히는지).
- 크롤러 파서: 동행복권 API 응답을 모킹(mock)해서 정상/비정상 응답 케이스를 테스트.
- 프론트엔드 E2E 테스트는 1단계 범위에서 제외한다 (속도 우선, 추후 필요 시 추가).

## 1단계 범위에서 제외하는 것

- 회원가입/로그인
- 구독/결제
- 광고 삽입
- 연금복권 데이터
- 타로점 등 추가 콘텐츠

이 항목들은 1단계 완료 후 별도의 설계 문서로 다룬다.
