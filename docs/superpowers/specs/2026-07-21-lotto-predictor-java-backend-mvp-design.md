# 로또 번호 예측 웹앱 — 1단계 MVP 설계 (v2: 프론트/백엔드 분리 아키텍처)

> 이 문서는 [2026-07-20-lotto-predictor-mvp-design.md](2026-07-20-lotto-predictor-mvp-design.md)를 대체한다. 배경/목적, "예측"에 대한 전제, DB 스키마의 기본 구조, 테스트 범위, 1단계 제외 항목 등은 그대로 유지하되, 아키텍처를 Next.js 단일 프로젝트에서 **Next.js 프론트엔드 + Java(Spring Boot) 백엔드 분리 구조**로 변경한다.

## 배경 및 목적

동행복권 사이트에 공개된 로또 6/45 역대 당첨번호를 자동으로 수집(크롤링)해서 DB에 쌓고, 이 데이터를 바탕으로 번호를 생성해주는 웹앱을 만든다. 이번 1단계 설계는 **로또 6/45만을 대상으로 한 핵심 기능 MVP**로 범위를 한정한다. 회원가입, 결제, 광고, 타로 운세 매칭 콘텐츠는 1단계에 포함하지 않는다 (타로 운세 매칭은 이 MVP 완료 후 별도 설계 문서로 다룬다).

### "예측"에 대한 중요한 전제

로또는 회차마다 완전히 독립적인 추첨이라, 과거 당첨 이력이 미래 당첨 확률에 통계적으로 영향을 주지 않는다. 이 앱의 "번호 생성"은 실제 예측력이 있는 것이 아니라, **출현 빈도 등 통계를 가중치로 삼은 랜덤 생성**이다. 사용자에게 오해를 주지 않도록 UI 문구에 유의한다 (예: "AI가 예측한 번호"보다는 "통계 기반 추천 번호").

## 아키텍처 개요

```
[Next.js 프론트엔드] --REST(JSON) 호출--> [Spring Boot 백엔드] --JDBC--> [Supabase Postgres]
     (Vercel 배포)                         (Render 무료 티어 배포)
```

- **프론트엔드**: Next.js(TypeScript). 페이지 렌더링만 담당하고, DB에 직접 접근하지 않는다. 모든 데이터는 Spring Boot REST API를 호출해서 가져온다.
- **백엔드**: Java + Spring Boot + Gradle. REST API 제공, 크롤링 로직, 번호 생성 로직, DB 접근(Spring Data JPA)을 모두 담당한다.
- **DB**: Supabase는 관리형 Postgres로만 사용한다. 백엔드는 Supabase의 REST/JS SDK를 쓰지 않고, 표준 JDBC 연결 문자열로 Spring Data JPA를 통해 직접 접속한다.
- **배포**: 프론트는 Vercel, 백엔드는 Render 무료 티어.

이 구성을 선택한 이유:
- 사용자가 Java로 백엔드를 작성하고 싶어 함.
- 프론트/백엔드 분리는 나중에 Capacitor 등으로 모바일 앱을 만들 때도 그대로 재사용 가능하다 — 앱은 그냥 같은 REST API를 호출하는 또 다른 클라이언트가 된다.
- Spring Boot 앱은 어디에 배포하든 코드가 동일해 특정 호스팅에 락인되지 않는다. Render 무료 티어로 시작하고, 트래픽이 늘면 Render 유료 플랜이나 다른 호스팅으로 손쉽게 옮길 수 있다.
- Render 무료 티어는 15분 유휴 시 슬립되고 재개 시 30~60초 콜드스타트 지연이 있다. 이 프로젝트는 크롤링이 수동 트리거이고 나머지 API도 트래픽이 적어, 이 지연을 감수할 수 있다고 판단했다.

## 저장소 구조

모노레포로 구성한다.

```
LottoPredictor/
├── frontend/   # Next.js (기존 스캐폴딩을 이 폴더로 이동, DB 직접 접근 코드는 제거)
└── backend/    # Spring Boot + Gradle (신규)
```

## 데이터베이스 스키마

`lotto_draws` 테이블 하나로 시작한다 (기존과 동일).

| 컬럼 | 타입 | 설명 |
|---|---|---|
| draw_no | integer, PK | 회차 번호 |
| draw_date | date | 추첨일 |
| num1 ~ num6 | smallint | 당첨번호 6개 |
| bonus_num | smallint | 보너스 번호 |
| created_at | timestamp | 저장 시각 (기본값: 현재 시각) |

`draw_no`가 기본키(유니크)이므로, 크롤러가 같은 회차를 다시 수집해도 upsert로 처리되어 중복 저장되지 않는다. 백엔드에서는 JPA 엔티티로 매핑한다.

## 크롤러

동행복권이 공개 제공하는 회차별 조회 API(`getLottoNumber`)를 이용해 데이터를 가져온다. HTML을 파싱하는 스크래핑 방식보다 안정적이다.

**자동 스케줄링 없이, 관리자가 수동으로 트리거하는 방식으로 진행한다.** 원래 검토했던 Vercel Cron / 주간 자동 실행은 이번 1단계에서 제외한다 — 새 회차가 나온 뒤 사용자가 원할 때 직접 실행하면 되고, 이렇게 하면 크론 인프라 없이도 충분하고 Render 무료 티어의 슬립도 문제가 되지 않는다.

동작 순서:
1. DB에서 저장된 가장 최신 회차 번호를 확인한다.
2. 그다음 회차부터 순서대로 데이터를 요청한다. (최초 실행 시에는 1회차부터 전체를 백필한다.)
3. 특정 회차 요청이 실패하거나 응답 형식이 예상과 다르면, 해당 회차는 건너뛰고 로그만 남긴다. 크롤러 전체가 중단되지 않으며, 다음 실행 때 다시 시도된다.
4. 아직 추첨되지 않은 회차에 도달하면(API가 "실패" 응답을 주면) 그 지점에서 멈춘다.
5. `draw_no`가 PK라 upsert로 처리되므로, 같은 트리거를 여러 번 눌러도 안전하다(멱등).

**트리거 방법**: 백엔드에 `POST /api/crawl` 엔드포인트를 두고, `Authorization: Bearer <secret>` 헤더로 보호한다. 프론트엔드에 "수집하기" 관리자 페이지를 만들어, 시크릿을 입력하면 브라우저가 이 엔드포인트를 직접 호출한다. 이 페이지는 별도 로그인 없이 시크릿 입력만으로 보호한다 (1단계는 회원 시스템이 없으므로).

## 번호 생성 로직

`GET /api/generate` 엔드포인트에서 처리한다.

1. DB에 저장된 전체 회차 데이터에서 1~45번 각 번호의 출현 빈도를 계산한다.
2. 출현 빈도를 가중치로 사용해, 가중 랜덤 샘플링(비복원 추출)으로 서로 다른 6개 번호를 뽑는다.
3. 가중치 없는 "완전 랜덤" 모드도 옵션으로 제공해, 사용자가 두 방식을 비교해볼 수 있게 한다.
4. DB에 데이터가 없는 경우에는 자동으로 완전 랜덤 모드로 대체(fallback) 동작한다.
5. 한 번에 여러 세트를 생성할 수 있다.

## 백엔드 API (Spring Boot)

| 엔드포인트 | 설명 | 보호 |
|---|---|---|
| `GET /api/generate` | 번호 생성 (가중/완전랜덤 모드, 세트 수 지정) | 공개 |
| `GET /api/stats` | 번호별 출현 빈도/확률 | 공개 |
| `GET /api/draws` | 역대 당첨번호 목록, 회차/날짜로 검색 | 공개 |
| `POST /api/crawl` | 최신 회차까지 크롤링 동기화 | `Authorization: Bearer <secret>` |

프론트엔드가 다른 도메인(Vercel)에서 호출하므로, 백엔드에 프론트 도메인을 허용하는 CORS 설정이 필요하다.

## 프론트엔드 페이지 구성

- **홈**: 번호 생성 버튼을 누르면 결과 세트가 표시된다. 여러 세트를 한 번에 생성할 수 있다.
- **통계**: 번호별 출현 빈도, 최근 오랫동안 나오지 않은 번호 등을 시각화해서 보여준다.
- **회차 조회**: 역대 당첨번호를 목록으로 보거나 회차/날짜로 검색할 수 있다.
- **수집하기(관리자용)**: 시크릿을 입력하고 버튼을 누르면 백엔드의 크롤링 동기화를 트리거한다. 결과(수집된 회차 수, 건너뛴 회차)를 화면에 보여준다.

## 기술 스택

- **백엔드**: Java, Spring Boot, Gradle, Spring Web, Spring Data JPA, JUnit5.
- **프론트엔드**: Next.js(TypeScript), App Router. DB 클라이언트(Supabase JS SDK 등)는 프론트에 두지 않는다.
- **DB**: Supabase(PostgreSQL, 무료 티어) — 관리형 Postgres로만 사용, JDBC로 접속.
- **배포**: 프론트 Vercel, 백엔드 Render(무료 티어).

## 에러 처리

- 크롤러의 개별 회차 실패는 건너뛰고 로그만 남기며, 전체 동기화를 중단시키지 않는다. 실패한 회차는 다음 수동 트리거 때 다시 시도된다(별도 상태 저장 없이 DB의 최신 `draw_no` 기준으로 재계산되므로 자연히 재시도된다).
- 번호 생성 시 DB가 비어있으면 완전 랜덤으로 자동 폴백한다.
- 동행복권 응답 형식이 변경되는 경우를 대비해, 파싱 실패는 예외로 처리하고 로그를 남긴다.
- 백엔드가 슬립 상태이거나 응답이 없을 때, 프론트엔드는 사용자에게 알아볼 수 있는 에러 메시지를 보여준다 (정확한 UI 문구/재시도 로직은 구현 단계에서 결정).

## 테스트 범위

- **백엔드(JUnit5)**: 가중 랜덤 번호 생성 로직의 분포 검증(특정 번호에 가중치를 높였을 때 실제로 더 자주 뽑히는지), 크롤러 파서를 모킹해서 정상/비정상 응답 케이스 테스트.
- **프론트엔드**: E2E 테스트는 1단계 범위에서 제외한다.

## 1단계 범위에서 제외하는 것

- 회원가입/로그인
- 구독/결제
- 광고 삽입
- 연금복권 데이터
- 타로점 등 추가 콘텐츠 (완료 후 별도 설계 문서)
- 자동 스케줄링 크론 (수동 트리거로 대체)
- 모바일 앱 패키징 (추후 Capacitor 등으로 같은 REST API를 재사용해 추가 가능, 별도 설계 문서)
